xcforge's UI automation tools (`find_element`, `tap`, `swipe`, `type_text`, `handle_alert`, etc.) require a **WebDriverAgent (WDA)** backend running on the target simulator. xcforge manages the WDA connection automatically — you just need to provide the project.

## How it works

xcforge supports two WDA backends, tried in order:

1. **xcforgeWDA** (preferred) — a lightweight fork of WebDriverAgent with xcforge's bundle ID. xcforge builds and launches it automatically if `XCFORGE_WDA_DIR` is set.
2. **Original WebDriverAgent** (fallback) — Facebook/Appium's WDA (`com.facebook.WebDriverAgentRunner`). xcforge falls back to this automatically if xcforgeWDA is unavailable and WDA is already running on port 8100.

You do not need Appium. xcforge talks to WDA directly over HTTP.

## Option A: xcforgeWDA (recommended)

xcforgeWDA is xcforge's own WDA fork. It uses the bundle ID `com.xcforge.wda.runner` to avoid conflicts with Appium installations.

### 1. Get the project

Clone the xcforgeWDA Xcode project and place it somewhere on your machine:

```bash
# Recommended: sibling to your xcforge install, or anywhere you prefer
git clone https://github.com/appium/WebDriverAgent.git xcforgeWDA
```

> **Note:** A dedicated `xcforgeWDA` repository is in progress. Until it's published, clone [appium/WebDriverAgent](https://github.com/appium/WebDriverAgent) and follow the bundle ID rename steps below.

After cloning, copy the Xcode project with the xcforge name and remap the bundle ID:

```bash
cp -r xcforgeWDA/WebDriverAgent.xcodeproj xcforgeWDA/xcforgeWDA.xcodeproj
sed -i '' 's/com\.facebook\.WebDriverAgentRunner/com.xcforge.wda.runner/g' \
  xcforgeWDA/xcforgeWDA.xcodeproj/project.pbxproj
```

Then add the `xcforgeWDARunner` scheme — copy `WebDriverAgentRunner.xcscheme` and update the two references inside it:

```bash
cp xcforgeWDA/xcforgeWDA.xcodeproj/xcshareddata/xcschemes/WebDriverAgentRunner.xcscheme \
   xcforgeWDA/xcforgeWDA.xcodeproj/xcshareddata/xcschemes/xcforgeWDARunner.xcscheme
sed -i '' 's/WebDriverAgent\.xcodeproj/xcforgeWDA.xcodeproj/g' \
  xcforgeWDA/xcforgeWDA.xcodeproj/xcshareddata/xcschemes/xcforgeWDARunner.xcscheme
```

### 2. Point xcforge at it

Set `XCFORGE_WDA_DIR` to the directory containing `xcforgeWDA.xcodeproj`.

**Claude Code / MCP (`.mcp.json`):**

```json
{
  "mcpServers": {
    "xcforge": {
      "command": "xcforge",
      "env": {
        "XCFORGE_WDA_DIR": "/path/to/xcforgeWDA"
      }
    }
  }
}
```

**CLI / shell:**

```bash
export XCFORGE_WDA_DIR=/path/to/xcforgeWDA
xcforge ui status
```

### 3. First run

xcforge builds and deploys xcforgeWDA automatically on the first UI tool call. Expect a 30–60 second delay on first use while it compiles. Subsequent calls reuse the cached build.

## Option B: Original WebDriverAgent (fallback)

If `XCFORGE_WDA_DIR` is not set and xcforgeWDA cannot be found, xcforge automatically falls back to any WDA already running on `localhost:8100` with the bundle ID `com.facebook.WebDriverAgentRunner.xctrunner`.

The most common way to get it running is via Appium:

```bash
npm install -g appium
appium driver install xcuitest
appium  # starts WDA on port 4723, which also starts WDA on the simulator
```

Or run it directly with xcodebuild after cloning [appium/WebDriverAgent](https://github.com/appium/WebDriverAgent):

```bash
xcodebuild build-for-testing \
  -project WebDriverAgent.xcodeproj \
  -scheme WebDriverAgentRunner \
  -sdk iphonesimulator \
  -derivedDataPath /tmp/wda-build

xcodebuild test-without-building \
  -xctestrun /tmp/wda-build/Build/Products/*.xctestrun \
  -destination "id=$(xcrun simctl list devices booted -j | python3 -c "import sys,json; d=json.load(sys.stdin); print(next(s['udid'] for sims in d['devices'].values() for s in sims if s['state']=='Booted'))")"
```

xcforge detects the running WDA and switches to the `originalWDA` backend automatically.

## Verify the connection

```bash
xcforge ui status --json
```

A healthy response includes the WDA version, session ID, and which backend is active (`xcforgeWDA` or `Original WDA`).

```bash
xcforge ui create-session --bundle-id com.example.MyApp --json
```

## Custom WDA URL

If WDA runs on a non-default port or host, override it:

**MCP:**
```json
"env": { "WDA_BASE_URL": "http://localhost:8200" }
```

**CLI:**
```bash
xcforge ui create-session --wda-url http://localhost:8200 --json
```

## Troubleshooting

| Symptom | Fix |
|---|---|
| `No WDA backend available` | Set `XCFORGE_WDA_DIR` or start Appium/WDA manually |
| Build fails during auto-deploy | Check Xcode is installed; run `xcodebuild -version` |
| WDA starts but tools time out | Booted simulator required; run `xcforge sim boot` first |
| `xcforgeWDA not available — using Original WDA` | `XCFORGE_WDA_DIR` is unset or points to wrong path |
