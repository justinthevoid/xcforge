This page focuses on repeatable, executable xcforge patterns.

## Pattern 1: Build and Test Loop

Use this loop when a change needs fast local validation.

```bash
xcforge defaults set --project MyApp.xcodeproj --scheme MyApp --simulator "iPhone 16 Pro"
xcforge build-test --json
xcforge test failures --include-console --json
xcforge test coverage --min-coverage 80 --json
```

Why this pattern works:

- `build-test` short-circuits early on compile failures.
- `test failures` returns structured failure context.
- Coverage checks add confidence before handoff.

## Pattern 2: Diagnosis Loop

Use this loop when you need a proof-oriented diagnosis trail. All action commands auto-resolve `--run-id` to the newest active run when omitted.

```bash
xcforge diagnose start --json
xcforge diagnose build --json
xcforge diagnose test --json
xcforge diagnose runtime --capture-screenshot --json
xcforge diagnose inspect --json
xcforge diagnose result --json
```

After applying a fix:

```bash
xcforge diagnose verify --json
xcforge diagnose compare --compact --json
```

## Pattern 3: Simulator and UI Automation Loop

Use this for flows that require app interaction and visual confirmation.

```bash
xcforge build run --json
xcforge ui alert --action accept_all --json
xcforge ui find --using "accessibility id" --value "Login" --scroll --json
xcforge screenshot capture --json
```

## Pattern 4: Evidence Capture Loop

Use this when you need logs and screenshots for verification.

```bash
xcforge log start --mode smart --json
# reproduce behavior
xcforge log read --include app --include crashes --json
xcforge screenshot baseline --name before-fix --json
# apply fix and rerun flow
xcforge screenshot compare --name before-fix --json
xcforge log stop --json
```

## Pattern 5: Hang Diagnostics

Use this when a build or test run is taking too long or timing out with no useful output.

**Default timeout is 180s.** On timeout, xcforge writes a diagnostic snapshot to `/tmp/xcf-diag-<pid>-<ts>.txt` containing a CPU sample, open file handles, and booted simulator state.

```bash
# Default — times out at 180s and surfaces a snapshot path + summary
xcforge build-test --json

# Long suites that legitimately take more than 3 minutes
xcforge build-test --long --json

# Capture a snapshot even on success, for performance baselining
xcforge build-test --diagnose --json
```

MCP equivalents:

```json
{ "long": true }
{ "diagnose": true }
```

Reading the snapshot: the `=== sample ===` section shows what the compiler or test runner was doing. The inline `Summary:` line in the tool result is the top 5 call frames.
